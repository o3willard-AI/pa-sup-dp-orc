## 2. Description

Build a type-safe configuration management system that loads, validates, and provides application settings from multiple sources (config files, environment variables, and command-line flags). This system will be the foundation for all runtime configuration throughout PairAdmin, enabling environment-specific deployments and user customization.

This task matters because configuration management is a cross-cutting concern that every other component will depend on. Getting this right early prevents technical debt and ensures consistent configuration patterns across the codebase.

2. **Define the main Config struct in `internal/config/config.go`**
   ```go
   package config
   
   type Config struct {
       App      AppConfig      `yaml:"app" json:"app"`
       LLM      LLMConfig      `yaml:"llm" json:"llm"`
       Terminal TerminalConfig `yaml:"terminal" json:"terminal"`
       Logging  LoggingConfig  `yaml:"logging" json:"logging"`
   }
   
   type AppConfig struct {
       Name        string `yaml:"name" json:"name"`
       Version     string `yaml:"version" json:"version"`
       DataDir     string `yaml:"data_dir" json:"data_dir"`

       Provider     string `yaml:"provider" json:"provider"`
   ```

4. **Implement configuration loading in `internal/config/config.go`**
   ```go
   // Load reads configuration from file, environment, and defaults
   func Load(configPath string) (*Config, error) {
       cfg := DefaultConfig()
       
       // Load from YAML file if provided
       if configPath != "" {
           if err := loadFromFile(cfg, configPath); err != nil {
               return nil, fmt.Errorf("failed to load config file: %w", err)
           }
       }
       
       // Override with environment variables
       loadFromEnv(cfg)
       
       // Validate configuration
       if err := validate(cfg); err != nil {
           return nil, fmt.Errorf("configuration validation failed: %w", err)
       }
       
       return cfg, nil
   }
   
   func loadFromFile(cfg *Config, path string) error {
       // Use gopkg.in/yaml.v3 for YAML parsing
       data, err := os.ReadFile(path)
       if err != nil {
           return err
       }
       return yaml.Unmarshal(data, cfg)
   }
   
   func loadFromEnv(cfg *Config) {
       // Override LLM API key from environment
       if envKey := os.Getenv(cfg.LLM.APIKeyEnvVar); envKey != "" {
           cfg.LLM.APIKey = envKey
       }
       // Add more env overrides as needed
   }
   
   func validate(cfg *Config) error {
       // Validate required fields
       if cfg.LLM.APIKey == "" && cfg.LLM.APIKeyEnvVar == "" {
           return errors.New("LLM API key must be provided")
       }
       // Add more validation rules
       return nil
   }
   ```

## 8. Constraints & Gotchas

**Known Pitfalls:**
- Do NOT use `viper` or other heavy configuration libraries - keep dependencies minimal for Phase 1
- Ensure all file paths use Go's `path/filepath` for cross-platform compatibility
- Remember that Windows uses different path separators than Unix
- The `~` home directory expansion must work cross-platform (use `os.UserHomeDir()`)

**Platform Considerations:**
- CGO is not required for this task (pure Go)
- Path handling must work on Windows, macOS, and Linux
- Environment variable names should be uppercase with underscores (standard convention)

**Interface Contracts:**
- The `Config` struct will be used by multiple packages - do not change field names after initial commit
- The `Load()` function signature must remain stable (may add options later via functional options pattern)
- All configuration errors must be wrapped with context using `fmt.Errorf` with `%w`

**Security Notes:**
- API keys should NEVER be stored in config files in production
- Always prefer environment variables for secrets (already built into design via `APIKeyEnvVar`)
- Add a comment in code warning against committing config files with real API keys

**Future-Proofing:**
- Design structs to allow easy addition of new configuration sections
- Keep validation logic separate from loading logic for testability
- Consider adding a `Validate()` method on Config for explicit validation calls
